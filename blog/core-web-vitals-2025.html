Core Web Vitals: Warum deine Website 2 Sekunden zu langsam ist

📅 18. Februar 2025 ⏱ 7 min Lesezeit ✍️ NoMoneyNoHoney.io

Jede Sekunde, die deine Website länger lädt, kostet dich Besucher, Rankings und Umsatz. Das ist keine Meinung — das sind Googles eigene Studiendaten. Und trotzdem haben die meisten Websites Core Web Vitals, die im roten Bereich hängen. Wir erklären, was LCP, CLS und INP wirklich bedeuten, wie du sie misst und — vor allem — wie du sie in 30 Tagen gezielt verbesserst.

Was sind Core Web Vitals — und warum sollte dich das interessieren?

Core Web Vitals sind Googles standardisierte Metriken für die Nutzererfahrung einer Website. Sie messen nicht, wie schön deine Seite aussieht, sondern wie sie sich anfühlt: Lädt sie schnell? Springt das Layout herum? Reagiert sie sofort auf Klicks?

Seit dem Page Experience Update sind Core Web Vitals ein direkter Google-Ranking-Faktor. Das bedeutet: Eine schlechte Performance kostet dich nicht nur Besucher, die abspringen — sie kostet dich auch Rankings. Gute SEO beginnt deshalb heute immer mit einer sauberen technischen Basis.

📊 Die Zahlen sprechen eine klare Sprache

Google-Studien zeigen: Jede 100ms LCP-Verbesserung erhöht die Conversion Rate im Schnitt um 8%. Bei E-Commerce-Websites mit 1 Mio. € Jahresumsatz = 80.000 € Mehrumsatz durch technische Optimierung allein.

LCP — Largest Contentful Paint: Warum deine Seite zu langsam lädt

LCP misst, wie lange es dauert, bis das größte sichtbare Element im Viewport geladen ist — typischerweise ein Hero-Bild, ein H1 oder ein Video. Google-Benchmark: unter 2,5 Sekunden. Alles über 4 Sekunden gilt als "schlecht".

Die häufigsten LCP-Killer: Nicht-optimierte Bilder (WebP statt JPEG), fehlende Preload-Hints für kritische Ressourcen, langsame Server-Antwortzeiten (TTFB über 600ms), blockierendes JavaScript und CSS.

CLS — Cumulative Layout Shift: Das unsichtbare Layout-Problem

CLS misst, wie stark sich deine Seite während des Ladens visuell verschiebt. Du kennst das: Du willst auf einen Button klicken, das Layout springt — und du klickst auf das Falsche. Frustrierend für Nutzer, teuer für Conversions. Google-Benchmark: unter 0,1.

Typische CLS-Ursachen: Bilder ohne definierte Dimensionen (width/height), nachträglich geladene Ads oder Embeds, dynamisch eingefügte Banner, Webfonts, die beim Laden das Layout verschieben. Die Lösung ist fast immer dasselbe: Platzhalterdimensionen für alle Elemente definieren, bevor sie geladen sind.

INP — Interaction to Next Paint: Die neue Interaktivitäts-Metrik

INP hat seit März 2024 FID (First Input Delay) abgelöst und misst die gesamte Reaktionszeit einer Seite auf Nutzerinteraktionen — nicht nur den ersten Klick. Benchmark: unter 200ms. Schlecht: über 500ms.

INP-Probleme entstehen meist durch schweres JavaScript, das den Main Thread blockiert. Long Tasks (über 50ms) sind der häufigste Schuldige. Lösung: JavaScript-Execution in kleinere Chunks aufteilen, Code-Splitting nutzen und nicht-kritisches JS verzögert laden.

Tools zum Messen — diese solltest du kennen

  • Google PageSpeed Insights — Kostenlos, zeigt Lab- und Field-Daten. Der erste Anlaufpunkt.
  • Lighthouse (in Chrome DevTools) — Detaillierter Audit direkt im Browser, ideal für Entwickler.
  • Google Search Console → Core Web Vitals Report — Zeigt echte Nutzerdaten aus deinem Traffic.
  • WebPageTest.org — Advanced Testing mit Filmstrip, Wasserfall und Global-Testing.
  • Chrome User Experience Report (CrUX) — Aggregierte Felddaten über echte Chrome-Nutzer.

💡 Wichtig zu verstehen

Lighthouse zeigt Lab-Daten (simuliert). Google Search Console zeigt Field-Daten (echte Nutzer). Für Rankings zählen die Field-Daten. Optimiere nach Lab — messe den Erfolg in Field.

In 30 Tagen besser werden — unser Prioritäts-Playbook

Woche 1 — Audit & Quick Wins: Bilder in WebP konvertieren, lazy loading aktivieren, Bild-Dimensionen definieren. Das allein verbessert LCP und CLS in den meisten Fällen messbar.

Woche 2 — Caching & Server: Browser-Caching konfigurieren, CDN einbinden, TTFB unter 200ms bringen. Bei WordPress: WP Rocket oder LiteSpeed Cache.

Woche 3 — JavaScript & CSS: Render-blocking Resources eliminieren, CSS Critical Path inlinen, nicht-kritisches JS defer/async laden. Code-Splitting bei React/Next.js implementieren.

Woche 4 — Fonts & Third-Party Scripts: Google Fonts self-hosten, Font-Display: swap setzen, Third-Party Scripts auditieren und nicht-kritische entfernen oder verzögern.

Unser Webentwicklungs-Team hat diesen Prozess in über 50 Website-Projekten verfeinert. Das Ergebnis: Ø 98 Lighthouse Score bei Auslieferung — keine leere Versprechen.

Fazit: Performance ist kein nettes Extra, sondern Grundlage

Core Web Vitals sind Googles Messung dafür, ob deine Website respektiert, dass Zeit der wertvollste Rohstoff deiner Besucher ist. Eine langsame Website sagt: "Deine Zeit ist mir egal." Eine schnelle Website sagt: "Ich bin bereit, wenn du es bist." Welche Botschaft willst du senden?
